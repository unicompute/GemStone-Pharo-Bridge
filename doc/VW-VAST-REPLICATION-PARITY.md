# VW/VAST Replication Parity Notes

This note records the current cross-reference against the sibling VisualWorks and VAST GemStone implementations for replication behavior, dirty-store traversal, and connector policy.

## Confirmed VisualWorks Semantics

The VisualWorks implementation treats `replicationSpec` as both a client-side replication policy and a source for clamped traversal metadata. The important operators observed in `GbsKernelBaseExt.st` are:

- `replicate`: materialize the slot.
- `stub`: exclude the slot from transitive closure.
- `min` and `max`: depth controls, also excluded from transitive closure specs.
- `forwarder`: leave the slot as a forwarder/proxy.
- `indexable_part`: policy for indexed slots in variable-sized objects.
- `callback`: use server-side callback policy rather than a static instvar-level table.

`GbxReplicationScheme >> transitiveClosureSpecForReplicationSpec:` now preserves the VW filtering rule: `min`, `max`, and `stub` entries are removed, while `replicate`, `forwarder`, and `indexable_part` entries remain. `#callback` produces an empty static transitive closure spec. Clamp-by-callback selector derivation also follows VW: each replication spec set uses `<replicationSpecSet>GbsClampCallback`, with `#transitiveClosureSpec` excluded.

## Implemented Pharo Coverage

- `GbsClientClassConnector` installs a converted replicator into the session replicator manager, registers the client/server pair, and updates clamp metadata from per-instvar replication specs. Reinstalling a connector for an already-mapped client or server removes the old replicator, secondary aliases, deferred server-instvar requests, clamp artefacts, and stale session connector pair before installing the replacement.
- `GbsServerClampSpecificationSynchronizer` installs a server `ClampSpecification` for live clamped traversal.
- `GbxReplicationScheme` validates clamp-by-callback selectors against live GemStone sessions via the mapped server behavior when available, with the original `Object canUnderstand:` path retained as the compatibility fallback before installing the selector into the server clamp specification.
- `GbsDirtyReplicateStoreTraversal` can use native dirty-store buffers and native `GciStoreTravDoTravRefs_` coordination for queued no-longer-replicated and GCed OOPs.
- Dirty-store traversal records a session-level write-path report showing native binary entries, semantic command entries, Dictionary/Set/Bag semantic counts, and custom semantic-policy counts for domain objects that opt into `gbsDirtyStoreRequiresSemanticCommand`. The live replication lane thresholds both native binary writes and semantic collection writes.
- VW-compatible object creation from free OOPs is available through `GbsSession>>createObjectClass:values:` and `createObjectClass:values:clusterBucket:permanent:`. These allocate a server object identifier with `GciGetFreeOop`, then create the object with `GciCreateByteObj` (byte-format classes, when `values` is a `ByteArray`) or `GciCreateOopObj` (pointer/Nsc classes), and answer a proxy to the new object. Pointer values may be proxies, already-mapped replicates, immediates, scalars resolved through native symbol/string OOP allocation, or forward references to other newly allocated free OOPs. The FFI binds `BoolType` as `int` (matching `gci.ht typedef int BoolType`), and `GbsServerInterface>>supportsObjectCreationTransport` gates availability on the 64-bit GCI transport.
- GCI FFI signatures were audited against the 3.7.5 `gci.hf`/`gci.ht` prototypes. Fixed: `GciFetchOops` now returns `int` with an `int numOops` argument (was `int64`/`int64`, mismatching `int GciFetchOops(OopType, int64, OopType[], int)`); `apiGciGetClass:` binds the real `GciFetchClass` (there is no `GciFetchClass_` symbol, so the prior underscore form resolved to a NULL `GCI_WEAK` pointer and only worked because the Core overlay rebinds the selector correctly); `GciIsKindOfClass` binds its OOP arguments as `uint64` rather than `void*`. The dead `apiGciIsString:` binding (non-existent `GciIsString` symbol, no callers) was removed. The `GciErrSType` manual field offsets (`exceptionObj`@16, `number`@104, `message`@113) were verified correct against the C struct with `GCI_MAX_ERR_ARGS=10`. `GciErr` retains its `bool` return because `fetchLastError` relies on a real Boolean result (`success ifFalse:`).
- Explicit VW-style export-set management is available through `GbsSession>>saveObjects:` / `releaseObjects:` (and `objectSpace saveObjects:`/`releaseObjects:`). `saveObjects:` resolves the given client objects/proxies/OOPs to server OOPs, cancels any pending no-longer-replicated release for them, and adds them to the gem's PureExportSet via `GciSaveObjs` so they survive in-gem garbage collection for the life of the session; `releaseObjects:` removes them again via `GciReleaseOops`. `GbsSessionLifecycleManager` records `savedOopCount`/`saveObjectsCallCount`/`releaseObjectsCallCount` counters (exposed in the lifecycle report), and `GbsServerInterface>>supportsObjectSaveTransport` gates availability on the 64-bit GCI transport. The `export-set-save-release` phase of `make replication-live` saves a materialized object, verifies it remains readable on the server, then releases it.
- Nested dirty-pool isolation follows VisualWorks `GbxDirtyPool>>setAsideDirtyWhile:`: the pool keeps a `dirtyStack`, and setting aside pushes the current dirty set, runs the block with a fresh empty top, flushes the nested set in its own pass if it is non-empty, then restores the previous set under `ensure:`. The modern materialized-object flush (`GbsDirtyReplicateStoreTraversal>>flush`) now runs `flushEntries:` inside `session setAsideDirtyWhile:`, so objects dirtied while the flush runs server execution (class-OOP resolution, new-element allocation, dictionary metadata, scalar-OOP resolution) are isolated from the flush's accounting and flushed separately rather than polluting or being lost from the in-progress set. The legacy `GbxStoreTraversal>>pack:` unmapped-class replicator path already used the same contract.
- `GbsSessionLifecycleManager` tracks session server maps, no-longer-replicated queues, GCed OOP queues, finalizer tokens, and release transport availability.
- `GbsSessionLifecycleManager` also records lifecycle counters for remembered/remapped OOPs, forgotten materialized objects, queued/acknowledged export-set OOPs, finalizer notifications, release-oops transport calls, and the last export-set flush summary. These counters are exposed through the session lifecycle report and thresholded in the live replication lane.
- Inter-session signal receipt now decodes the full signal triple. `InterSessionSignal` carries `signalNumber`, `sentMessage`, and `sendingSessionSerial` (with `messageText` falling back to the decoded message), mirroring VisualWorks `GsInterSessionSignal` accessors. `GbsSession>>gemSignalAction:` adds VW's three-argument gem-to-gem handler, evaluated with `(sendingSerial, signalNumber, message)` for the first signal and each polled successor; the existing one-argument `sessionSignalAction:` proxy path is retained. With no handler installed, the default path raises an `InterSessionSignal` enriched with the decoded triple, falling back to the raw error text when the signal cannot be decoded. The triple is fetched in a single server round trip via `Object _objectForOop:`/`signalFromSession` reading `sendingSessionSerial`/`sentInt`/`sentMessage`.
- `GbxReplicatorManager` now treats the replicator as the lifecycle unit for client aliases, server aliases, deferred server-instvar updates, and clamp artefacts, so secondary client mappings and server removals clean up consistently.
- `GbxReplicatorManager` exposes VW-style client-class connector compatibility entrypoints plus a structured connector-manager report covering client/server map sizes, connector-pair count, connected/disconnected session pair counts, post-connect actions, deferred server-instvar work, active behavior names, replication-spec set, and clamp count. The launcher includes this report next to the replication policy and lifecycle reports.
- The GemBuilder settings UI (`GbsSettingsPresenter`) surfaces the replication-policy templates: a template picker (default/production/migration/debug from `GbsReplicationPolicy templateNames`) with an Apply-template button that populates the replication controls from `templateNamed:`, and a Show-diff button that reports the current form's policy diff from defaults (`diffFromDefaultsText`). The picker logic is factored into headless-testable helpers (`applyReplicationTemplateNamed:`, `currentReplicationPolicyFromPresenters`, `replicationDiffText`). The form now renders every `GbsReplicationPolicy` setting spec, including `serverMapLeafCapacity` (with a matching `GbsConfiguration` accessor). The new explicit export-set save/release counters (`savedOopCount`/`saveObjectsCallCount`/`releaseObjectsCallCount`) appear in the launcher's Session Lifecycle Report, since the lifecycle report text iterates all lifecycle counter keys.
- Per-instance-variable clamp levels are editable through `GbsClampLevelEditorPresenter`, which surfaces the `GbxReplicationScheme`/`GbxClampSpecification` `instVarLevels` model as an instVar-name -> level table. It builds one row per named instance variable (aligned to the class's `allInstVarNames`, with the currently stored level), and writes edited levels back through `replaceInstVarLevelsForClass:with:` (clearing the class's clamp levels when all rows are unset, and trimming trailing unset slots). It can be opened directly (`openOnScheme:class:`) or resolved from a live session and client class (`openForSession:clientClass:`). The row-building, level-array round-tripping, and write-back logic are headless-testable (`clampLevelRows`, `levelsArrayFromRows:`, `applyEditedRows:`).
- `GbsClassConnector` connect now registers the concrete connector object before manager installation, so connector subclass type, session connection state, and post-connect action survive the Pharo `GbsClientClassConnector` bridge. Direct replicator-manager removal also unregisters session connector pairs and connection-state registry entries, matching the connector disconnect lifecycle.
- `GbxClampManager` now updates callback metadata, named replication schemes, class-shape/class-version refreshes, and removal cleanup across all known schemes instead of only the currently active scheme. Named schemes ask the client class for the selected replication-spec selector when it exists, so `#customerReplicationSpec` and `#callbackReplicationSpec` no longer silently reuse only the default `#replicationSpec`, including when the selector is inherited.
- Explicit materialized-object stubbing and fault refresh are available through session/object-space APIs. Stubbing evicts materialized identity caches, unregisters dirty/server-map state, optionally frees local slots according to `freeSlotsOnStubbing`, and queues no-longer-replicated OOPs only for real local replicates. Faulting through a proxy fetches a fresh replicate and cancels a pending no-longer-replicated release for that OOP.
- `make replication-live` validates connector install, connector-manager report counts, server clamp synchronization, callback clamp selector synchronization, dynamic server callback traversal for a live migration-domain object, per-instvar clamp entries, inherited named replication specs, a same-shape live migration-domain order/customer fixture with forwarder/min/max/stub/nil slot policy, remapped live migration-domain fixtures with VisualWorks-style `instVarMap` server-to-client offsets, client-only `nil` mapped slot mutation that does not dirty the replicate, inherited `super instVarMap , ...` subclass extension behavior, mapped class-version refresh of server offsets and clamp levels, native named-slot dirty writes for all domain fixtures, mapped lifecycle commit/abort/logout/reconnect cleanup, clamped traversal, native dirty-store flush, lifecycle queued/acknowledged export-set counters, a twelve-object business-write dirty traversal fixture, export-set queue acknowledgement, native Set/Bag dirty writes whose unmapped elements are created in-flush via the `GCI_STORE_TRAV_CREATE` flag, VW-parity free-OOP object creation of a byte object and a forward-referencing pointer object via `GciCreateByteObj`/`GciCreateOopObj`, and trend thresholds against a live GemStone session.

## Remaining Parity Watchpoints

- Broader migration-shaped live callback coverage is still useful for real application domain classes, but the built-in order/customer fixture now validates a real `<replicationSpecSet>GbsClampCallback` method whose returned levels depend on the server object's state and are fetched through the callback clamp scheme.
- Per-instvar traversal semantics are live-validated for a five-slot same-shape order fixture, a five-slot remapped order fixture with an unmapped server slot, an inherited remapped subclass fixture, and mapped class-version refresh. Strict application parity should still be checked against domain-specific `instVarMap` methods from migration projects.
- VAST `faultLevelRpc` compatibility is represented in the policy surface, but deeper fault transition behavior still needs migration-shaped live tests.
- The native dirty-store buffer covers mapped Arrays, Associations, OrderedCollections, named-slot objects, Dictionaries (via per-association native reports), and Sets/Bags with OOP-addressable elements. Sets and Bags that gain brand-new, unmapped element objects now also flush natively: each unmapped trackable element is allocated a free OOP and emitted as its own new-object report (`buildNativeStructuralEntriesForHashedCollectionEntry:`), and the traversal buffer sets the `GCI_STORE_TRAV_CREATE` (0x2) flag so GciStoreTrav creates those objects from their free OOPs in the same flush instead of failing with error 2101. This is validated by the `unmapped-element-dirty-store` phase in `make replication-live`, which asserts a native flush, zero semantic Set/Bag fallback, two free-OOP allocations, and server-side creation of the added elements. A Set/Bag whose unmapped elements cannot be tracked/class-resolved still falls back cleanly to the semantic command path (`canNativeStoreHashedCollectionEntry:`). Custom/domain objects needing richer semantic policy can still force the semantic command path with `gbsDirtyStoreRequiresSemanticCommand`.
- A complete VisualWorks replicator-manager clone may still expose deeper inherited server callback behavior and connector subclass policy edge cases, but the Pharo manager now covers the core connector alias, replacement, removal, deferred-instvar, class-version refresh, selector-driven named scheme, and multi-scheme clamp lifecycle semantics.
- The VW connector subclasses are now ported from skeleton stubs to working models: `GbsClassVarConnector` (client/server class-variable names `cvarName`/`gsvarName` with the VW fallback to `cvarName`, `stvarName`/`variableName`, `stName:gsName:cvarName:` / `stName:gsName:stvarName:gsvarName:` constructors, and `connectsAssociation` true), `GbsClassInstVarConnector` (connects a class-instance-variable value, so `connectsAssociation` is false and `variableKind` is `#classInstanceVariable`), and `GbsFastConnector` (binds `clientObj` to a server `serverKernelOop` by identity via `stObj:gsObj:`, resolving the OOP from an integer or any object answering `#oop`). Covered by `GbsConnectorSubclassTest`.
- Native FFI audit (3.7.5 headers / macOS Security framework). Verified clean: the keychain bindings (`SecKeychainAddGenericPassword`, `SecKeychainFindGenericPassword`, `SecKeychainItemDelete`, `SecKeychainItemFreeContent`, `SecKeychainItemModifyAttributesAndData`, `CFRelease`) all match the canonical prototypes (`UInt32`→`uint`, `OSStatus`→`int`, pointers→`void*`), and the call sites marshal the `UInt32*` password length into a 4-byte buffer read back with `uint32At:`. `GciClampedTravArgsSType` is laid out correctly (40 bytes: clampSpec@0, resultOop@8, travBuff@16, level@24, retrievalFlags@28, isRpc@32). KNOWN ISSUE: `GbsStoreTravDoTravRefsArguments` declares `GciStoreTravDoArgsSType` as 104 bytes with `storeTravBuff`/`alteredTheOops`/`storeTravFlags` at byte offsets 80/88/96, but the struct is actually 96 bytes with those fields at 72/80/88 (clang-verified against `gcicmn.ht`). The malformed struct makes `GciStoreTravDoTravRefs_` fail, so the dirty-store path falls back to plain `GciStoreTrav` plus a separate manual release -- correct results in two trips, which is why the lane stays green. Correcting the struct activates the real single-trip `DoTravRefs_` path, which then exposes a separate latent bug: combining no-longer-replicated/GCed release OOPs with a Bag NSC store in one trip corrupts the Bag write (the VW-documented hazard of releasing an object in the same round trip that stores into it). Fixing the struct therefore requires also splitting store-from-release (or guarding Bag/NSC stores) before the single-trip path can be enabled; the struct correction is intentionally not shipped on its own.
